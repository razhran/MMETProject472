مرجع شامل لبنية الـ backend، نماذج البيانات، الـ RBAC، والـ offline sync

Node.js 18+ Express 4 SQLite (WAL) JWT HS256 bcrypt 12r PWA Offline-First

نظرة عامة على النظام

جدول users — مع حقل is_active للبوابة اليدوية

is_active: عندما يسجّل مستخدم جديد يُحفظ بـ is_active = 0 و role = 'pending'. لا يمكنه الدخول حتى يُغيّر المدير الرئيسي هذا الحقل يدوياً عبر PATCH /api/admin/users/:id/activate. 
CREATE TABLE users (
  id           TEXT    PRIMARY KEY,                      -- UUID v4
  username     TEXT    NOT NULL UNIQUE  COLLATE NOCASE, -- case-insensitive
  email        TEXT    NOT NULL UNIQUE  COLLATE NOCASE,
  password     TEXT    NOT NULL,                          -- bcrypt hash, 12 rounds

role         TEXT    NOT NULL DEFAULT ‘pending’
CHECK(role IN (‘pending’,‘club’,‘editor’,‘super_admin’)),

– ▼▼▼ ACTIVATION GATE — المفتاح الرئيسي للنظام ▼▼▼
is_active    INTEGER NOT NULL DEFAULT 0,              – 0=pending, 1=active
– ▲▲▲ يتغير فقط بواسطة المدير الرئيسي ▲▲▲

club_name    TEXT,                                      – اسم النادي للعرض
club_id      TEXT,                                      – ID للفلترة (sports, tech…)
login_count  INTEGER NOT NULL DEFAULT 0,              – إحصائية
last_login   TEXT,
created_at   TEXT    NOT NULL DEFAULT (strftime(’%Y-%m-%dT%H:%M:%SZ’,‘now’)),
updated_at   TEXT    NOT NULL DEFAULT (strftime(’%Y-%m-%dT%H:%M:%SZ’,‘now’))
);

دورة حياة المستخدم (State Machine)

  Registration ──▶  { role: 'pending', is_active: 0 }
                            │
              PATCH /admin/users/:id/activate
              (super_admin ONLY — Raed)
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
        { role: 'club'  { role: 'editor' { role: 'super_admin'
          is_active: 1 }  is_active: 1 }  is_active: 1 }
              │
   PATCH /admin/users/:id/deactivate  ──▶  { is_active: 0 }
   (immediate: all refresh tokens revoked)

Middleware — auth.js

يحتوي على ثلاثة أجزاء: إنشاء التوكن، التحقق منه، وحراسة الأدوار.

دالة requireAuth()

// تسلسل التحقق:
1. استخراج 'Bearer TOKEN' من Authorization header
2. jwt.verify(token, JWT_SECRET, { algorithms: ['HS256'] })
   → TokenExpiredError  ──▶  401 { code: 'TOKEN_EXPIRED' }   // api.js يعيد المحاولة بـ refreshToken
   → JsonWebTokenError  ──▶  401 { code: 'TOKEN_INVALID' }
3. SELECT FROM users WHERE id = payload.id
   → user not found     ──▶  401 { code: 'USER_NOT_FOUND' }
   → is_active === 0    ──▶  403 { code: 'ACCOUNT_INACTIVE' }   // لحظي عند التعطيل
4. req.user = { id, username, role, club_name, club_id }
   next()

نظام RBAC — طبقات الصلاحيات

المبدأ: كل route تُطبّق عليها صلاحية بشكل صريح في ملف الـ route. لا يوجد "وضع افتراضي آمن" — كل طلب يجب أن يمر بـ guard صريح. 
// RBAC Hierarchy
super_admin  ←──  أعلى صلاحية (رائد الزهراني)
     ↑
editor       ←──  يمكنه نشر كل أنواع المحتوى
     ↑
club         ←──  يمكنه نشر أخبار نادية فقط (category='club')
     ↑
pending      ←──  لا يملك أي صلاحية حتى التفعيل

// Guards المتاحة:
requireAuth()          // أي access token صالح
requireClub()          // club | editor | super_admin
requireEditor()        // editor | super_admin
requireSuperAdmin()    // super_admin ONLY
requireRole(…roles)  // factory — أي تركيبة مخصصة

Club RBAC Enforcement في news.controller.js

if (user.role === 'club') {
  category   = 'club';          // مقيّد — لا يمكن تغييره من الطلب
  club_id    = user.club_id;    // ناديه فقط — لا يمكن نشر باسم نادٍ آخر
  authorName = user.club_name || user.username;  -- اسم النادي تلقائياً
} else {
  // editors/super_admins: حرية اختيار category
  category = allowed.includes(req.body.category) ? req.body.category : 'general';
}

Auth Controller — منطق الأعمال

register()

// POST {AUTH_PREFIX}/register
1. express-validator التحقق من المدخلات
2. فحص تعارض username/email (استعلام واحد)
3. await bcrypt.hash(password, 12)
4. INSERT users { role: 'pending', is_active: 0 }   ← دائماً
5. INSERT notifications → target_role: 'super_admin'   ← يُبلَّغ رائد
6. audit log: REGISTER
7. 201 { message: 'Registration submitted' }   ← لا توكن بعد

login()

// POST {AUTH_PREFIX}/login
1. SELECT * FROM users WHERE username=? OR email=?
2. // Constant-time comparison — لمنع timing attacks
   const DUMMY = '$2b$12$INVALID.HASH.PADDED.FOR.CONSTANT.TIME...';
   const match = user ? bcrypt.compare(pw, user.password)
                      : bcrypt.compare(pw, DUMMY); ← يعمل حتى لو لا يوجد مستخدم
3. فحص role === 'pending' ──▶  403 PENDING_APPROVAL
4. فحص is_active === 0    ──▶  403 ACCOUNT_INACTIVE
5. issueTokenPair(user, { ip, userAgent })
   ├── accessToken:  JWT HS256, expires 15m
   └── refreshToken: 48-byte random hex → sha256 hash → DB
6. UPDATE users SET last_login, login_count++
7. 200 { accessToken, refreshToken, expiresAt, user: safeUser }

refresh()

// POST {AUTH_PREFIX}/refresh — بدون requireAuth (لأن access token منتهٍ)
1. sha256(refreshToken) → البحث في refresh_tokens
2. فحص expires_at > now()
3. جلب المستخدم — فحص is_active
4. DELETE refresh_tokens WHERE id = stored.id   ← Token Rotation
5. issueTokenPair(user) — توكن جديد كلياً
6. 200 { accessToken, refreshToken, expiresAt }

Cache Controller — Offline Sync

هذا المتحكم هو حلقة الوصل بين الـ backend والـ Service Worker. يُنتج snapshot واحداً يحتوي على كل البيانات العامة التي تحتاجها الصفحات.

getSnapshot()

// GET /api/cache/snapshot — public
// Cache-Control: public, max-age=300, stale-while-revalidate=60

return {
generatedAt: now,
news:          SELECT … WHERE is_published=1 LIMIT 30,
events:        SELECT … WHERE event_date BETWEEN now AND now+30days,
missing:       SELECT … WHERE is_approved=1 AND is_resolved=0,
ads:           SELECT … WHERE is_active=1 AND now BETWEEN start_at AND end_at,
notifications: SELECT … WHERE target_role=‘all’
}
// لا يحتوي على: passwords, tokens, emails, private data

getDelta()

// GET /api/cache/delta — lightweight check
// الـ PWA يتحقق هل البيانات تغيّرت قبل تحميل snapshot كامل

return {
news:          MAX(updated_at) FROM news,
events:        MAX(updated_at) FROM events,
missing:       MAX(created_at) FROM missing_items,
ads:           MAX(updated_at) FROM ads,
notifications: MAX(created_at) FROM notifications
}

Offline-First Sync Strategy

الهدف: المستخدم العادي لا يشعر بفرق إذا انقطع الإنترنت. يعرض البيانات المحفوظة تلقائياً.

مسار البيانات الكامل

─── عند أول تشغيل (أو بعد كل 5 دقائق) ──────────────────────

api.js: Cache.forceRefresh()
──▶ GET /api/cache/snapshot
──▶ localStorage.setItem(‘madkhal_snapshot’, JSON.stringify({ data, ts }))

─── عند طلب البيانات ──────────────────────────────────────────

api.js: API.getNews(category, page)
├── try: fetch(’/api/news?category=X’)
│     ✓ success ──▶ return response
│     ✗ network error (offline):
│         └── Cache.get() → localStorage
│               ✓ found: return { data: cached.news, offline: true }
│               ✗ not found: throw error
└── ui shows toast: ‘يعرض بيانات محفوظة’

─── Service Worker (sw.js) ────────────────────────────────────

App Shell (HTML/CSS/JS):  Cache-First strategy
├── hit: serve from STATIC_CACHE instantly
└── miss: fetch → cache → return

/api/cache/snapshot:      Network-First + DATA_CACHE fallback
├── network ok: serve + update cache
└── offline:    serve from DATA_CACHE

/api/* (other):           Network-Only
└── offline: return { success: false, offline: true }

─── عند العودة للإنترنت ───────────────────────────────────────

window.addEventListener(‘online’, () => {
Cache.forceRefresh()  // تحديث صامت في الخلفية
document.dispatchEvent(new CustomEvent(‘mk:online’))
})

🧪 اختبار الـ API

اختبار التسجيل والدخول

# تسجيل حساب نادي جديد
curl -X POST http://localhost:3000/my-secret-gate-2026/register \
  -H "Content-Type: application/json" \
  -d '{"username":"sports_club","email":"sports@jic.edu","password":"Test@1234","club_name":"النادي الرياضي","requested_role":"club"}'

# محاولة الدخول قبل التفعيل — يجب أن يعود 403 PENDING_APPROVAL
curl -X POST http://localhost:3000/my-secret-gate-2026/login   
-H “Content-Type: application/json”   
-d ’{“username”:“sports_club”,“password”:“Test@1234”}’

# دخول المدير الرئيسي
curl -X POST http://localhost:3000/my-secret-gate-2026/login   
-H “Content-Type: application/json”   
-d ’{“username”:“raed_admin”,“password”:“MyStrongPass@123!”}’

تفعيل حساب (بتوكن المدير)

# احتفظ بالـ accessToken من خطوة دخول المدير
TOKEN="eyJ..."
USER_ID="uuid-of-sports-club"

curl -X PATCH http://localhost:3000/api/admin/users/$USER_ID/activate   
-H “Authorization: Bearer $TOKEN”   
-H “Content-Type: application/json”   
-d ’{“role”:“club”,“club_name”:“النادي الرياضي”,“club_id”:“sports”}’

اختبار RBAC — نشر خبر بحساب نادٍ

# الدخول بحساب النادي بعد التفعيل
CLUB_TOKEN="eyJ..."

# محاولة نشر خبر كـ category=‘general’ — سيُغير تلقائياً إلى ‘club’
curl -X POST http://localhost:3000/api/news   
-H “Authorization: Bearer $CLUB_TOKEN”   
-F “title=انتصار النادي الرياضي”   
-F “body=
فاز نادينا بالبطولة!
”   
-F “category=general”
# ← ستجد في الـ response: category=‘club’ بغض النظر عما أرسلته

# جلب الـ snapshot (public — بدون توكن)
curl http://localhost:3000/api/cache/snapshot | python3 -m json.tool

# فحص الـ delta
curl http://localhost:3000/api/cache/delta
